bioseq::io::detail::run_sync_pipeline_mmap_direct

Synchronous pipeline specialised for

Synchronous pipeline specialised for mmap_source. The whole file is already in contiguous virtual memory; we hand the parser a single span over the mapped region instead of cycling 64 KiB slabs through read(2) + memmove. Wins:

  • Zero memcpy on the fast path. The slab in run_sync_pipeline copies every byte once via mmap_source::read_into's std::memcpy(dst, _addr + _cursor, copy_size). That's pure overhead — the parser only reads bytes, never writes them.

  • No make_room memmove. For a single-record file (e.g. the bench's 234 MiB human-genome corpus), the slab path slides a ~48 KiB tail back to the slab's start every refill cycle. The direct path skips this entirely.

  • No slab grow. The slab's geometric grow chain (64 KiB → 128 KiB → … → ~8 MiB) is irrelevant when the file's already mapped.

  • The parser sees is_eof = true on the first call, so the FSM emits the trailing record cleanly without a "wait for more" dance.

The reservation hint for the batch's _seq_storage uses the file size: dna4 packs 4 letters per byte, so ~file_size / 32 u64 words bound the storage requirement. Pre-reserving this avoids the ~9 geometric reallocs that would otherwise fire while encoding a 234 MiB body.
